## 1.1 问题所在：上下文熵增 (Context Entropy)

### 什么是上下文熵增？

在长时间的 AI 辅助编程会话中，随着对话轮次增加，模型的上下文会逐渐"漂移"——早期的重要信息被后期的无关信息淹没，导致模型性能下降。

### 典型症状

```
Turn 1-10:    模型完全理解项目结构和目标
Turn 11-30:   偶尔需要提醒关键约束
Turn 31-50:   开始忘记早期决策，出现矛盾建议
Turn 51+:     可能需要重置会话，重新开始
```

### 现有解决方案及其局限

| 方案 | 原理 | 局限 |
|------|------|------|
| **RAG** | 从外部知识库检索 | 检索质量不稳定，延迟高 |
| **摘要** | 定期压缩历史 | 信息丢失，累积误差 |
| **滑动窗口** | 只保留最近 N 轮 | 完全丢失早期上下文 |
| **CLAUDE.md** | 项目级记忆文件 | 静态，无法自动更新 |

### 练习：体验上下文漂移

**任务**：在一个 50 轮的编程会话中跟踪以下指标：
- 模型重复提问的次数
- 与早期决策矛盾的建议次数
- 用户需要澄清/纠正的次数

**思考**：如果有一种方式能让 AI "记住"整个项目的演变历史，而不需要每次都重新加载？

## 1.2 从被动到主动：范式转移

### "始终在线"计算的历史演进

```
1980s: 屏幕保护程序（防止 CRT 烧屏）
1990s: 系统托盘程序（后台通知）
2000s: 推送通知系统（智能手机）
2010s: 智能助手（Siri, Google Now）
2020s: 环境式 AI（Ambient AI）← 我们在哪里
```

### 为什么守护进程模式是自然终局？

编程不是离散的任务序列，而是持续的认知过程：
- 你在思考时，代码在变化
- 你在休息时，问题在发酵
- 你在编码时，上下文在不断累积

一个只在被调用时才"存在"的助手，永远无法真正理解这个过程。

### KAIROS 所处的技术交汇点

```
         持久计算                    低成本推理                 信任架构
       (云端+本地)                  (模型效率)                (权限+沙箱)
            │                          │                        │
            └────────────┬─────────────┴────────────┬───────────┘
                         │                          │
                         ▼                          ▼
              ┌─────────────────────┐
              │   KAIROS: 持久自主   │
              │   AI 编程助手        │
              └─────────────────────┘
```

## 1.3 希腊渊源：为什么叫 "KAIROS"？

### Chronos vs. Kairos

| 概念 | Chronos (Χρόνος) | Kairos (Καιρός) |
|------|------------------|-----------------|
| 含义 | 物理时间、可量化 | 恰当时机、质性的 |
| 比喻 | 钟表滴答 | 箭在弦上 |
| 关注点 | "多久" | "何时" |

### 设计启示

**基于 Chronos 的代理**：
```python
# 每隔 N 秒检查一次
while True:
    if should_act():
        act()
    sleep(INTERVAL)
```

**基于 Kairos 的代理**：
```python
# 等待机会窗口
while True:
    tick = wait_for_tick()
    if is_opportune(tick) and has_high_value_action():
        act_within_budget()
    else:
        sleep()  # 尊重用户注意力
```

### 15 秒预算的哲学

这不是一个随意的数字。它是：
- **可感知的**：用户不会觉得被阻塞
- **足够的**：完成有意义的快速任务
- **受限的**：强制智能体保持专注

这是关于**尊重用户注意力**的具体体现。

### ✅ 任务 1: 计算你的上下文熵增（难度 ⭐）

**目标**: 量化"会话漂移"现象

**操作**:
1. 打开一个你最近工作的代码项目
2. 查看 git log，找到最近一个 2 小时以上的编程会话
3. 记录：
   - 会话开始时的目标：_________
   - 会话结束时的实际完成：_________
   - 中间"走神"去做的事：_________（如：修了个不相关的 bug、重构了无关文件）

**计算**:
```
上下文熵增 = (走神次数 / 会话时长) × 100%
```

**问题**: 如果你的 AI 助手能在你走神时提醒你，你会希望它在什么时机介入？

### ✅ 任务 2: 绘制你的 Chronos vs Kairos 时间线（难度 ⭐）

**目标**: 理解两种时间哲学的区别

**操作**:
取昨天的工作/学习，绘制两条时间线：

**Chronos 时间线**（钟表时间）：
```
09:00 ─── 10:30 ─── 12:00 ─── 14:00 ─── 17:00
  │         │          │          │         │
开会      写代码     吃饭      写代码     下班
```

**Kairos 时间线**（恰当时机）：
```
"灵感闪现" ──→ "进入心流" ──→ "遇到卡点" ──→ "豁然开朗"
    │              │              │              │
  10:15         10:45          14:30          16:00
```

**问题**: 
- 你的心流时段通常持续多久？
- 什么时机打断你最不造成破坏？
- AI 助手应该在"遇到卡点"时介入吗？
